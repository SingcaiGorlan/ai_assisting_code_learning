## ⚠️ 核心问题

### 架构冲突
```
❌ 项目同时包含两种架构：

1. Wails 桌面应用（web/wails-app/）
   └─ Go bindings 通信

2. Web 应用（internal/app/handler/）
   └─ HTTP REST API 通信

导致：
- 前端通信方式混乱
- 后端职责不清
- 构建流程冲突
```

---

## 🎯 关于 Element Plus

### 问题
> "可否将前端更换成 Element Plus 解决？"

### 答案：**❌ 不建议**

### 原因

| 维度 | 说明 |
|------|------|
| **迁移成本** | 110-170 小时（3-4 周） |
| **代码重写** | 100% 前端代码需要重写 |
| **当前状态** | React + Radix UI 已无错误，运行良好 |
| **收益** | 有限，不如解决架构问题 |
| **建议** | 保持现有技术栈 |

### 更好的选择
- ✅ 保持 React + Radix UI + Tailwind
- ✅ 如需更多组件：添加 shadcn/ui 或 MUI
- ✅ 优先解决架构冲突

---

## 🔧 解决方案

### 方案 A: Wails 桌面应用（推荐用于桌面场景）

**适合**: 需要真正的桌面应用、访问系统资源、离线使用

**行动**:
1. 移除 Gin HTTP 后端
2. 业务逻辑改为 Wails methods
3. 前端改用 Wails bindings
4. 删除 HTTP 相关代码

**时间**: 1-2 天

---

### 方案 B: 纯 Web 应用（推荐用于 Web 场景）

**适合**: 需要浏览器访问、多用户在线、服务器部署

**行动**:
1. 移除 Wails 配置
2. 保留 Gin 后端作为 API 服务器
3. 前端独立部署为 SPA

**时间**: 0.5-1 天
